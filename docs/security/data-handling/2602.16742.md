# Data Handling - 2602.16742 (DeepVision-103K)

## Summit Readiness Assertion

This data-handling profile inherits governance from `docs/SUMMIT_READINESS_ASSERTION.md` and deny-by-default security policy.

## Scope

Rules for ingesting and evaluating `2602.16742` artifacts and related benchmark outputs inside Summit.

## Data Classification

- Public:
  - Paper text, benchmark tables, model/dataset metadata, published links.
- Internal:
  - Evaluation configs, intermediate metrics, CI logs.
- Confidential:
  - Raw downloaded dataset samples and local evaluation transcripts.
- Restricted:
  - Secrets, API tokens, credentials, and any user-identifying data discovered during processing.

## License and Source Constraints

The paper lists mixed-license sources, including `Unset` entries.

- Default: block import until source license and usage constraints are explicitly approved.
- Allowlist required per source before any dataset mirror or transformation.
- Preserve source attribution in evidence metadata.

## Never-Log Rules (Hard)

- Never log secrets, access tokens, credentials, or cookies.
- Never log raw proprietary dataset payloads to persistent CI artifacts.
- Never log full prompt/response bodies when they can contain licensed or sensitive data.
- Never store PII fields unredacted.

## Retention Rules

- Repo-stored evidence is limited to deterministic artifacts:
  - `report.json`
  - `metrics.json`
  - `stamp.json`
- Raw dataset caches must remain outside git and be treated as ephemeral.
- Timestamps are allowed only in `stamp.json`.

## Access Control

- Deny-by-default for dataset import and external connectors.
- Explicit policy grant required for each new source.
- Human approval required for any policy exception.

## Evidence and Audit Requirements

- Every run must include a valid `evidence_id` (`EVD-*`).
- Schema validation required against `evidence/*.schema.json` or `summit/evidence/schemas/*.json`.
- Hash integrity should be included in evidence index entries when available.

## Threat Model Alignment

- Threats:
  - Data leakage through logs/artifacts.
  - License non-compliance from mixed-source ingestion.
  - Metric tampering via non-reproducible outputs.
- Mitigations:
  - Never-log + redaction gates.
  - License allowlist policy fixtures.
  - Deterministic evidence verifier in CI.

## MAESTRO Security Alignment

- MAESTRO Layers: `Data`, `Tools`, `Observability`, `Security`.
- Threats Considered: leakage, policy bypass, compliance drift.
- Mitigations: deny-by-default access, schema-validated evidence, CI gate enforcement.
