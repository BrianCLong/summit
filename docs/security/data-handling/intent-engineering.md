# Intent Engineering Data Handling Policy

## Classification and Retention

This policy defines handling requirements for intent-engineering artifacts generated by Summit validation workflows.

- Classification: internal governance telemetry.
- Default retention: 30 days for generated reports unless superseded by stricter legal hold requirements.
- Storage scope: artifact stores approved by repository governance controls.

## Prohibited Data in Logs and Artifacts

Do not persist:

- Raw user prompts.
- Secrets, API keys, tokens, credentials, or bearer headers.
- Full external content bodies not required for deterministic validation evidence.

## Required Data Hygiene

- Prefer hashed intent identifiers for cross-run analysis.
- Minimize payload footprint in artifacts to fields necessary for determinism and policy verdicts.
- Use explicit allowlists for any field serialized to JSON reports.

## Runtime Constraints

- Validation steps operate in no-external-call mode unless explicit allowlisting exists.
- Unknown constraint types fail closed.
- Constraint violations must produce machine-readable failure reasons.

## Artifact Contract

The intent validation flow emits:

- `artifacts/intent/report.json`
- `artifacts/intent/metrics.json`
- `artifacts/intent/stamp.json`

`stamp.json` is the only approved location for non-deterministic execution metadata.

## Compliance Mapping

This policy aligns with repository governance requirements for:

- Evidence-first outputs.
- Reversible, explainable automated decisions.
- CI-enforced policy controls.
