# Summit Knowledge Graph Data Model

The Summit knowledge graph is the core data structure underpinning the Agentic AI OSINT Platform. It provides a deterministic, verifiable chain-of-custody for open-source intelligence. The data model is strictly typed and immutable by default to ensure integrity.

## Entity Type Taxonomy

Nodes in the knowledge graph represent discrete entities. The current supported taxonomy of node types is defined as `NodeType`:

*   **`actor`**: Represents an individual, organization, or automated bot operating on a platform.
    *   *Description*: A participant capable of generating content, establishing relationships, or amplifying signals.
    *   *Examples*: A user profile on X, a public channel on Telegram, a registered corporation.
*   **`content`**: Represents a discrete piece of information generated by an actor.
    *   *Description*: Textual, structured, or composite data payloads that form the primary signals analyzed by the platform.
    *   *Examples*: A post, a comment, an article, a public message.
*   **`url`**: Represents a web resource reference.
    *   *Description*: Extracted links that serve as indicators of information flow or external references.
    *   *Examples*: `https://example.com/article`, shortened links.
*   **`media`**: Represents non-textual or binary assets.
    *   *Description*: Images, videos, audio clips, or documents associated with content or actors.
    *   *Examples*: An attached photo, a linked PDF, a video file.
*   **`topic`**: Represents a high-level subject, theme, or extracted concept.
    *   *Description*: Abstract nodes used to cluster content, actors, or other entities based on semantic similarity or categorical labels.
    *   *Examples*: `#disinformation`, `cybersecurity`, `geopolitics`.

*Note*: Actors, content, and media can be associated with a specific `Platform` (`x`, `facebook`, `telegram`, `other`).

## Relationship Type Catalog

Edges in the knowledge graph represent directed relationships between nodes. The supported edge types are defined as `EdgeType`:

*   **`engages`**
    *   *Directionality*: Actor → Content / Actor
    *   *Cardinality*: Many-to-Many
    *   *Description*: A direct interaction with a target entity, such as liking, commenting, or replying.
*   **`mentions`**
    *   *Directionality*: Content → Actor / Topic / URL
    *   *Cardinality*: Many-to-Many
    *   *Description*: An explicit reference to another entity within a piece of content.
*   **`amplifies`**
    *   *Directionality*: Actor → Content
    *   *Cardinality*: Many-to-Many
    *   *Description*: An action that increases the visibility of content, such as a retweet or a share.
*   **`co_shares`**
    *   *Directionality*: Actor → URL / Media
    *   *Cardinality*: Many-to-Many
    *   *Description*: When multiple actors distribute the exact same URL or media asset.
*   **`crossposts`**
    *   *Directionality*: Actor / Content → Content / Platform
    *   *Cardinality*: Many-to-Many
    *   *Description*: The act of publishing identical or highly similar content across multiple platforms or channels.

## Node Property Schemas

Nodes are implemented as frozen dataclasses to ensure immutability once instantiated.

**Required Properties:**
*   `id` (`str`): A globally unique identifier for the node.
*   `type` (`NodeType`): The strictly typed category of the entity (`actor`, `content`, `url`, `media`, `topic`).
*   `attrs` (`dict[str, Any]`): A schemaless dictionary for storing node-specific attributes (e.g., metadata, extracted features, raw JSON payloads).

**Optional Properties:**
*   `platform` (`Optional[Platform]`): The platform where the entity originated (`x`, `facebook`, `telegram`, `other`). Can be `None` for platform-agnostic entities like topics.

## Edge Property Schemas

Edges are implemented as frozen dataclasses, representing a directed link at a specific point in time.

**Required Properties:**
*   `src` (`str`): The `id` of the source node.
*   `dst` (`str`): The `id` of the destination node.
*   `type` (`EdgeType`): The strictly typed relationship category.
*   `ts` (`int`): A Unix timestamp indicating when the relationship occurred or was observed.

**Optional/Defaulted Properties:**
*   `weight` (`float`, defaults to `1.0`): A numerical value representing the strength, confidence, or magnitude of the relationship.
*   `attrs` (`Optional[dict[str, Any]]`, defaults to `None`): A dictionary for edge-specific metadata (e.g., interaction duration, specific contextual flags).

## Entity Lifecycle

As an OSINT platform requiring strict verifiable chain-of-custody, data mutation is highly controlled.

1.  **Creation**: Entities (Nodes and Edges) are instantiated as frozen dataclasses during the ingestion or extraction pipelines. Once created in memory, their state cannot be mutated.
2.  **Update**: Because entities are frozen, "updates" are typically handled by appending new state. If a node's attributes must change, a new node instance is created with the updated attributes, and the system records the transition to maintain provenance.
3.  **Deprecation/Deletion**: Hard deletions are generally avoided to preserve the intelligence audit trail. Instead, nodes or subgraphs may be marked as deprecated (e.g., via a specific flag in `attrs` or tracked via the broader architecture's `lifecycle` state) to exclude them from active queries while retaining the historical record.

## Graph Constraints and Invariants

To guarantee the reliability of the knowledge graph:

*   **Immutability**: `Node` and `Edge` instances are strictly frozen. Attempting to modify `id`, `type`, `attrs`, or `ts` post-instantiation will raise an error.
*   **Strict Typing**: Node and Edge types are restricted to the `NodeType` and `EdgeType` literals. Dynamic creation of unapproved types is blocked at the code level.
*   **Referential Integrity**: An `Edge` must reference valid `src` and `dst` IDs. While the dataclass itself does not validate the existence of the IDs in the broader graph, graph ingestion and orchestration layers enforce that edges cannot point to non-existent nodes.
*   **Temporal Ordering**: Every edge must have a valid integer timestamp (`ts`). Graph traversals and historical queries rely on these timestamps to reconstruct the state of the network at any given point in time.

## Data Lineage Tracking Model

Ensuring deterministic, verifiable chain-of-custody is a fundamental requirement of the Summit platform.

*   **Raw Data Anchoring**: Every node and edge must trace back to a verifiable source. The raw payload or extraction evidence is stored or referenced within the `attrs` dictionary.
*   **Provenance Metadata**: The platform tracks the `source`, `timestamp`, `actor` (system or user), and specific `evidence` (such as hashes of original artifacts) for operations that mutate the logical state of the graph.
*   **Attribution in GraphRAG**: When the GraphRAG system retrieves information, it provides granular node-level attribution. Every evaluated claim outputs specific `supporting_nodes`, `contradicting_nodes`, and flags `fabricated_entities`, ensuring that AI-generated insights are always grounded in the verifiable graph data model.
