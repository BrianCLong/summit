# Runbook: Required-Check Drift

## Symptoms

- "Green-but-stuck" PRs where all visible checks pass, but auto-merge is blocked because GitHub is waiting on a required check context that will never arrive.
- CI saturation delays while developers debug missing checks.
- CI workflows failing early due to the Sentinel drift detection.

## What the Sentinel Checks

The `Required-Check Drift Sentinel` periodically compares:

1.  **Required Contexts**: The authoritative list of required status checks for the `main` branch, retrieved directly from the GitHub API.
2.  **Produced Contexts**: The set of check contexts actually being generated by the CI pipeline and external integrations.
    -   *Dynamic*: Contexts fetched from recent commits on `main` (both check runs and commit statuses).
    -   *Static*: Job names and workflow names parsed directly from `.github/workflows/*.yml`.

If there is a required context that is not found in either the dynamic or static produced sets, it triggers a **Drift** alarm. This prevents regressions from misconfigured CI triggers or renamed jobs that break branch protection rules.

## How to Reproduce Locally

You can manually trigger the sentinel script locally using a Personal Access Token (PAT) with `repo` permissions to read branch protection.

1.  Export your GitHub token:
    ```bash
    export GITHUB_TOKEN=your_pat
    ```
2.  Run the drift sentinel script:
    ```bash
    node scripts/ci/check_required_contexts_drift.mjs
    ```
    *(Optional)* Run in static-only mode to ignore recent check-run history:
    ```bash
    node scripts/ci/check_required_contexts_drift.mjs --static-only
    ```

## Fix Playbook

If the sentinel script reports missing required contexts, classify them to apply the correct fix:

### Scenario 1: Missing "Workflow / Job" (Likely Workflow Rename/Trigger Issue)

If the missing context looks like a CI job name (e.g., `Unit Tests`, `Build`, `Release Ops / Smoke Test`), a recent commit likely renamed the workflow file, renamed the job, removed the job, or altered the trigger condition.

**Fix Steps:**
1.  **Restore the Job Name**: If the job was intentionally renamed but the branch protection rule was not updated, revert the job name or update the `.github/workflows/*.yml` file to match the exact string required by the branch protection rules.
2.  **Verify Triggers**: Ensure the workflow has `pull_request` triggers defined if the job is supposed to run on PRs. Sometimes PR triggers are accidentally removed or incorrectly restricted to specific paths.
3.  **Matrix Jobs**: If the missing context corresponds to a matrix expansion, ensure that all dimensions of the matrix are correctly yielding the expected context string.
4.  Re-run the sentinel locally to confirm the fix, and validate on a test PR.

### Scenario 2: Missing External Context (Likely App/Webhook Integration)

If the missing context belongs to a third-party application (e.g., `codecov/project`, `license/cla`), the integration might have failed, uninstalled, or is misconfigured.

**Fix Steps:**
1.  Confirm that the corresponding GitHub App is installed and authorized for the repository.
2.  Check the third-party service's dashboard for errors, billing issues, or configuration drifts (e.g., missing secrets).
3.  Verify that webhooks are successfully delivering payloads from GitHub to the third-party service.
4.  If the service is permanently deprecated, coordinate with a repo admin to remove the context from the required checks list in the repository settings.

## Rollback Notes

If the Sentinel itself introduces false positives blocking legitimate CI changes, you may temporarily disable the workflow by reverting the PR that introduced it. **Do not** modify branch protection settings unless specifically instructed to do so as part of a formal governance change.
