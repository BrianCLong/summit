# ADR-002: Choice of Vector Store

## Status
Accepted

## Context
Semantic retrieval is a core component of Summit's GraphRAG pipeline. We need a vector store that can store and query high-dimensional embeddings for entities, documents, and sub-graphs. The requirements include low-latency retrieval for real-time applications and robust, persistent storage for long-term intelligence data, all while maintaining strict tenant isolation.

## Decision
We have decided to adopt a **hybrid vector store architecture** using **Redis** and **pgvector (PostgreSQL)**.

1. **Redis**: Used as a high-performance cache for query embeddings and frequently accessed document chunks. Redis provides the ultra-low latency required for the "hot" path of retrieval.
2. **pgvector (PostgreSQL)**: Used for persistent storage of entity and document embeddings. PostgreSQL's `pgvector` extension allows us to store vectors alongside relational metadata, ensuring transactional integrity and easy tenant scoping via existing RBAC/ABAC models.

## Consequences
- **Positive**: Combines the speed of in-memory caching (Redis) with the reliability and relational power of PostgreSQL (pgvector). Simplifies deployment by leveraging databases already in use (PostgreSQL).
- **Negative**: Requires managing two different vector storage mechanisms and ensuring consistency between the cache and the primary store.
- **Neutral**: Performance tuning (e.g., indexing strategies like HNSW or IVFFlat) will be required as data volume grows.
