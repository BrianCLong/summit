# ADR-005: Ingestion Pipeline Architecture

## Status
Accepted

## Context
Summit must ingest a diverse range of data sources, from high-volume streaming OSINT feeds to periodic batch uploads of structured intelligence reports and human intel. The system requires a unified ingestion framework that can handle different data formats, ensure data quality, capture lineage, and maintain strict provenance while scaling to meet demand.

## Decision
We have adopted a **hybrid ingestion architecture** that supports both **Batch (ETL)** and **Real-time (Streaming)** data flows through a unified set of normalization and validation components.

1. **Batch Pipeline**: Optimized for large-scale data processing (CSV, JSON, GPX) using scheduled ETL jobs (Airflow/Temporal). This path is designed for high throughput and deep historical data loads.
2. **Streaming Pipeline**: Powered by **Kafka/Redpanda** for real-time ingestion of live signals and event-driven data. This path is optimized for low-latency enrichment and immediate availability in the graph.
3. **Unified Normalization**: Both paths share a common set of normalization, validation, and H3 indexing services to ensure consistent data quality and schema adherence regardless of the ingestion method.
4. **Provenance Manifests**: All ingestion events (batch or streaming) generate cryptographically signed manifests to ensure tamper-evident provenance and a clear chain of custody.

## Consequences
- **Positive**: Provides the flexibility to handle any data source while maintaining high standards for data quality and lineage. Shared components reduce duplication of logic and simplify maintenance.
- **Negative**: Increased complexity in managing both streaming and batch infrastructures. Requires careful coordination to ensure data consistency between parallel ingestion paths.
- **Neutral**: Requires clear monitoring and dead-letter queue (DLQ) management for both paths to ensure reliability.
