# ADR-004: LLM Provider Abstraction Layer Design

## Status
Accepted

## Context
The LLM landscape is rapidly evolving, with new models and providers emerging frequently. Summit needs to be resilient to provider outages, avoid vendor lock-in, and allow for the use of specialized models (e.g., highly secure models for sensitive data or lightweight models for summarization) depending on the specific task. Hard-coding provider-specific logic across the codebase is unsustainable.

## Decision
We have implemented a unified **LLM Provider Abstraction Layer** (centered around the `LLMIntegration` component).

Key design features:
1. **Generic Interface**: Standard methods for generation, embedding, and summarization that are agnostic to the underlying provider (OpenAI, Anthropic, Azure OpenAI, local models, etc.).
2. **Adapter Pattern**: Specialized adapters for each provider that translate between Summit's internal formats and the provider's API.
3. **Dynamic Routing**: The ability to route requests to specific models or providers based on task type, cost limits, or required security clearance.
4. **Resiliency**: Support for fallback models and retries across different providers.

## Consequences
- **Positive**: Eliminates vendor lock-in, increases system resilience, and allows us to easily benchmark and adopt new models. Centralizes cost tracking and token usage monitoring.
- **Negative**: Adds a small amount of overhead for the abstraction layer. Requires maintaining multiple adapters as provider APIs change.
- **Neutral**: Developers must use the abstracted interface instead of calling provider SDKs directly.
