# ADR-006: API Authentication Mechanism

## Status
Accepted

## Context
Summit is deployed across diverse environments, from public cloud SaaS offerings to highly secure, air-gapped government networks. We need an authentication mechanism that is flexible enough to support human analysts, automated services, and federated nodes, while meeting the stringent security requirements of the intelligence community (including MFA, OIDC, and mTLS).

## Decision
We have decided on a **multi-tiered authentication strategy** managed by the API Gateway:

1. **JWT (JSON Web Tokens) with OIDC**: Used for primary user authentication. JWTs contain user identity, roles, and scopes, allowing for stateless authorization at the service level. Integration with OIDC/SSO providers ensures centralized identity management.
2. **API Keys**: Used for service-to-service authentication and external integrations. API keys are cryptographically generated, scoped to specific roles, and support automatic rotation.
3. **mTLS (Mutual TLS)**: Mandatory for communication between core system components and federated nodes in high-security environments. This ensures both encryption and cryptographic verification of the identity of all communicating parties.
4. **Step-up Authentication**: Sensitive operations (e.g., data export, administrative changes) require phishing-resistant WebAuthn step-up authentication.

## Consequences
- **Positive**: Provides a defense-in-depth security posture with appropriate authentication for different use cases. Supports a wide range of deployment scenarios.
- **Negative**: Adds configuration and management complexity for multiple authentication methods. Requires robust secret management for API keys and certificates.
- **Neutral**: Authentication logic is centralized in the API Gateway to simplify backend service development.
